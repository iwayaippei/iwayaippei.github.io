# ソフトウェア工学 第1回～第6回まとめ

## 第1回 ガイダンス

第1回では、本講義の目的や進め方について学んだ。本講義では、ソフトウェアエンジニアリングの基礎知識を理解するとともに、実務で活用できるスキルを身に付けることを目標としている。また、ソフトウェア工学の考え方を学び、実際の開発現場でどのように活用されているかを考える機会を提供することが講義の狙いである。

ソフトウェア工学は、実際のソフトウェア開発の経験から体系化された学問であり、知識を学ぶだけでは十分ではなく、実践を通して身に付けることが重要である。また、個人の技術だけでなく、チーム全体で知識や情報を共有しながら開発を進めることの重要性についても学んだ。

さらに、講師の経験をもとに、システム更改やオープンソースソフトウェアの開発、Rパッケージの開発などの事例が紹介された。ソフトウェアは開発して終わりではなく、保守やテスト、ライブラリの依存関係への対応など継続的な管理が必要であることを理解した。また、近年ではAIエージェントや大規模言語モデル（LLM）を利用したソフトウェア開発が進んでいるが、AIを活用する場合でも要件定義やバージョン管理、テストなどのソフトウェア工学の知識は不可欠であり、人間が開発全体を管理・監督する能力が重要であることを学んだ。

## 第2回 ソフトウェア工学概論

第2回では、ソフトウェア工学の基本的な考え方について学んだ。ソフトウェアとは、プログラムだけでなく、データや利用方法を説明する文書も含めたものである。ハードウェアのように物理的に劣化することはないが、技術や利用環境の変化に対応するため、継続的な保守や改良が必要になる。

ソフトウェア工学とは、ソフトウェアの開発・運用・保守を体系的かつ効率的に行うための学問であり、「品質（Quality）」「コスト（Cost）」「納期（Delivery）」の3つをバランスよく実現することが目的である。また、問題を理解し、設計し、実装し、テストによって品質を確認するという一連の流れがソフトウェア開発の基本であることを学んだ。

さらに、ソフトウェアの大規模化やIT人材不足、社会インフラとしての重要性の高まりなどから、効率的な開発や保守を実現するためにソフトウェア工学が必要であることを理解した。また、将来の機能追加や仕様変更に対応しやすいように、保守性や拡張性、再利用性を考慮した設計を行うことが重要であると学んだ。

## 第3回 ソフトウェアライフサイクル

第3回では、ソフトウェアライフサイクルについて学んだ。ソフトウェアは企画・計画から始まり、要件定義、設計、実装、テスト、運用・保守を経て、最終的には廃止されるまでの一連の流れを持つ。そのため、開発だけではなく、運用や保守まで含めて考えることが重要である。

要件定義では、システムの目的や必要な機能、性能、制約条件などを明確にし、開発する機能と開発しない機能を整理することが重要である。設計では要件を具体的な仕様へ落とし込み、実装可能な設計書を作成する。その後、設計書に従ってプログラムを作成し、テストによってバグを発見・修正する。すべてのバグを完全になくすことは難しいため、効率よく品質を高めることが重要である。

また、運用・保守では、システムを継続的に改善し、利用状況を分析しながら機能追加や修正を行うことが求められる。複数人での開発では、仕様書や設計書などのドキュメントを整備し、情報共有を行うことが開発効率や品質向上につながることを学んだ。

## 第4回 プロジェクトマネジメントと開発プロセス

第4回では、プロジェクトマネジメントと開発プロセスについて学んだ。プロジェクトとは、期限が決められた独自の目標を達成するための活動であり、目標から逆算して計画を立てるバックキャスティングの考え方が重要である。目標を明確に設定し、それを達成するための作業を整理することで、効率的な開発につながる。

また、品質・コスト・納期（QCD）のバランスを考えながらプロジェクトを進めることが重要であり、システムの種類によって何を優先するかが異なることを学んだ。さらに、ソフトウェアの規模や品質は、コード量やバグ発生率、ファンクションポイント法などを用いて評価できることも理解した。

開発プロセスとしては、ウォーターフォール型、スパイラルモデル、反復型開発、アジャイル開発について学んだ。ウォーターフォール型は工程ごとに順番に進めるため管理しやすい反面、仕様変更への対応が難しい。一方、アジャイル開発は短いサイクルで開発と改善を繰り返し、変化する要求に柔軟に対応できる。それぞれに長所と短所があり、プロジェクトの目的や規模に応じて適切な開発手法を選択することが重要である。

## 第5回 WBS

第5回では、WBS（Work Breakdown Structure）について学んだ。WBSとは、プロジェクト全体を細かな作業単位へ分解し、階層的に整理することで、プロジェクト全体を管理しやすくする手法である。作業内容や担当者、スケジュールを明確にできるため、進捗管理や役割分担を効率よく行うことができる。

WBSを作成する際には、まずプロジェクトのスコープを明確にし、バックキャスティングの考え方を用いて大きな作業を分類する。その後、各作業をさらに細かく分解し、作業同士の順序や関連性を整理する。作業の漏れや重複がないように整理するMECEの考え方も重要である。

演習ではゲーム大会の開催を例として、企画から準備、当日の運営、片付けまでのすべての作業を洗い出し、それぞれの担当者や開始日・終了日を設定することで、実際のプロジェクト管理の流れを学んだ。

## 第6回 WBS演習

第6回では、第5回に引き続きWBSを用いた演習を行った。WBSは単に作業を一覧化するだけではなく、プロジェクト全体を可視化し、各メンバーの担当範囲や進捗状況を共有するための重要なツールであることを学んだ。

また、プロジェクトは計画どおりに進まない場合も多いため、WBSを活用して進捗を定期的に確認し、必要に応じて計画を修正することが重要である。WBSを適切に運用することで、作業漏れやスケジュールの遅れを防ぎ、チーム全体で効率よくプロジェクトを進められることを理解した。

第1回から第6回までの講義を通して、ソフトウェア開発ではプログラミング技術だけではなく、計画立案、設計、品質管理、チームでの情報共有、プロジェクト管理が重要であることを学んだ。今後の演習や実際の開発では、これらの知識を活用し、より効率的で品質の高いソフトウェア開発を実践していきたい。

# ソフトウェア工学（第1～13回）を通して学んだこと

本講義を通して、ソフトウェア開発は単にプログラムを書く作業ではなく、多くの人が協力しながら品質の高いソフトウェアを継続的に開発・運用するための仕組みや考え方が重要であることを学んだ。開発工程や要件定義、設計、実装、テストといった一連の流れを理解することで、それぞれの工程に目的があり、どれか一つでも疎かにすると品質や保守性に大きな影響を与えることを理解した。

また、GitやGitHubを利用したバージョン管理について学び、コミットやブランチ、マージ、プルリクエストなどの操作を実際に行うことで、複数人で効率よく開発を進める方法を身に付けることができた。これまではプログラムを保存する程度の認識しかなかったが、変更履歴を管理したり、開発内容を共有したりするための重要なツールであることを実感した。

さらに、CI/CDの考え方について学び、GitHub Actionsを利用することで、ソースコードの更新からビルドや公開までを自動化できることを理解した。手作業で行っていた作業を自動化することで、作業効率の向上だけでなく、人的ミスの防止や品質の維持にもつながることを学んだ。GitHub PagesとJekyllを用いたホームページの公開では、Markdownで作成した文書が自動的にWebページへ変換される仕組みを体験し、CI/CDが実際の開発現場でどのように活用されているのかを理解することができた。

講義全体を通して特に印象に残ったのは、ソフトウェア開発では技術だけでなく、チーム開発を円滑に進めるためのルールや開発プロセスが非常に重要であるという点である。個人でプログラムを書く場合とは異なり、複数人で開発を行うためには、共通の開発手順やバージョン管理、レビュー、テストなどを適切に行うことが品質の向上につながることを学んだ。

今回学んだ内容は、今後の情報工学総合演習や実践的システム開発演習だけでなく、将来ソフトウェア開発に携わる際にも役立つ知識であると感じた。プログラミングの技術を高めるだけではなく、開発全体を意識しながら設計や管理、運用まで考えられる技術者を目指したい。また、GitやGitHub、CI/CDなどのツールを積極的に活用し、効率的で品質の高いソフトウェア開発を実践できるよう、今後も継続して学習を続けていきたい。
